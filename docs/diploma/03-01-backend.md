# ГЛАВА 3. РЕАЛИЗАЦИЯ И ТЕСТИРОВАНИЕ

## 3.1 Реализация backend-части на Django

Backend-часть платформы реализована на Django 5.1 и Django REST Framework. Django выбран как зрелый web-фреймворк, предоставляющий ORM, систему миграций, встроенную модель пользователя, административную панель и удобные механизмы настройки проекта. Django REST Framework используется для реализации HTTP API, через которое взаимодействуют frontend, checker и внешние клиенты.

Структура backend располагается в каталоге `backend/`. Основная предметная логика находится в приложении `ctf`. Модели описаны в файле `backend/ctf/models.py`, API endpoint-ы — в `backend/ctf/views.py`, маршруты — в `backend/ctf/urls.py`. Такой подход соответствует типовой структуре Django-проекта и упрощает навигацию по коду.

Конфигурация проекта находится в Django-пакете `config`. Backend запускается в контейнере Docker и получает параметры подключения к PostgreSQL, secret key, debug-режим, список разрешенных host-ов, CORS origins и адреса demo vulnbox-сервисов через переменные окружения. Для локальной демонстрации это важно: один и тот же код можно запускать с настройками по умолчанию из `docker-compose.yml` или переопределять значения через `.env`.

Предметная модель реализована через Django ORM. Миграции фиксируют таблицы команд, участников, сервисов, раундов, флагов, отправок, статусов сервисов, настроек соревнования и заявок на резервирование. Ограничения уникальности задаются на уровне моделей и базы данных: уникальны названия и slug команд, названия и slug сервисов, номера раундов, значения флагов, а также пары `team-service-round` для `Flag` и `ServiceStatus`. Благодаря этому часть инвариантов предметной области контролируется не только кодом API, но и хранилищем данных.

Для аутентификации используется token authentication. После успешного входа пользователь получает token, который frontend сохраняет на клиентской стороне и передает в последующих запросах. Endpoint `/auth/me/` возвращает текущий пользовательский контекст: данные пользователя, команду, роль, состав команды и историю последних отправок.

Регистрация команд реализована с учетом настроек соревнования. Backend проверяет, открыта ли регистрация, не истекло ли окно регистрации и требуется ли предварительное резервирование. В зависимости от настроек новая команда может быть сразу одобрена или отправлена на модерацию. Это позволяет использовать платформу как для открытых учебных тренировок, так и для мероприятий с ручным контролем состава участников.

Во время регистрации backend создает капитана, дополнительных участников и записи `TeamMember` внутри транзакции. Перед созданием проверяется уникальность username в переданном составе, максимальный размер команды, уникальность названия и slug команды. Если включен режим резервирования, token заявки проверяется на статус `approved`, срок действия и соответствие выбранным `team_name` и `team_slug`. После успешной регистрации заявка переводится в `claimed`, что исключает повторное использование token.

Капитанские операции реализованы отдельными endpoint-ами. Функция проверки капитана возвращает текущего пользователя, членство и ошибку доступа; это уменьшает дублирование в `team_update`, `team_add_member`, `team_update_member_role` и `team_remove_member`. Backend не позволяет удалить собственную учетную запись через API управления командой и не допускает ситуацию, при которой в команде не остается ни одного капитана. Эти проверки относятся к бизнес-логике, поэтому они выполняются на backend-е независимо от состояния frontend-интерфейса.

Логика отправки флагов реализует несколько защитных проверок. Система не засчитывает неизвестный флаг, повторную сдачу уже принятого флага, истекший флаг и флаг собственной команды. Такие проверки важны для честного Attack/Defense scoring. Каждая попытка сохраняется в истории `Submission`, поэтому администратор и участники могут видеть не только успешные, но и отклоненные действия.

Endpoint отправки флага работает только для участника активной одобренной команды и только при наличии running round. Перед поиском флага применяется rate limit: не более десяти отправок от команды за минуту. Значение флага ищется в таблице `Flag` внутри транзакции с блокировкой строки через `select_for_update`. Это снижает риск гонки, при которой один и тот же флаг мог бы быть принят несколькими параллельными запросами. При успешной проверке создается `Submission` со статусом `accepted` и начисляется 25 атакующих баллов; при ошибке создается `rejected` запись с поясняющим сообщением.

Административная часть API защищена staff-проверками. Через нее организатор может управлять настройками регистрации, заявками, командами, сервисами и раундами. Для раундов реализованы операции создания, пакетного планирования, запуска, завершения и генерации недостающих флагов. Endpoint checker tick используется как вручную из панели администратора, так и автоматически из отдельного checker-контейнера.

Реализация раундов вынесена частично в модуль `backend/ctf/competition.py`. Функция `ensure_flags_for_round` создает флаги для всех активных одобренных команд и активных сервисов. Значение флага формируется в виде строки `BALTCTF{...}` с номером раунда, идентификаторами команды и сервиса и случайным token. Это удобно для демонстрации: по формату видно, что флаг относится к платформе, но угадать конкретное значение невозможно из-за случайной части.

Функция `run_checker_tick` реализует backend-часть игрового цикла. Она требует, чтобы раунд был в состоянии `running`, создает недостающие флаги, собирает активные команды и сервисы, выполняет service-specific checker modules и сохраняет результаты через `ServiceStatus.objects.update_or_create`. Ошибки checker-а преобразуются в доменные статусы `down`, `mumble` и `corrupt`, а успешная проверка дает `up`. Для каждого статуса вычисляются защитные баллы по методу `ServiceStatus.suggested_points`.

Для dashboard, scoreboard и service status backend формирует расширенные агрегированные ответы. Они включают атакующие и защитные баллы, количество отправок, статистику accepted/rejected, breakdown статусов сервисов, последние отправки и последние checker-события. Это снижает сложность frontend-компонентов: большая часть предметных расчетов выполняется на сервере, где доступны все данные и можно гарантировать единые правила.

Агрегация реализована в наборе внутренних функций `views.py`: строятся summary, scoreboard, service stats, round stats, service status history и admin state. Scoreboard не хранится в отдельной таблице. Для каждой команды backend суммирует accepted `Submission` как атакующие баллы и `ServiceStatus.points_awarded` как защитные баллы, а затем сортирует команды по общему результату и дополнительным метрикам. Такой подход упрощает контроль корректности: если изменяется история отправок или checker-статусов, итоговые показатели пересчитываются из источников истины.

Для подготовки демонстрационной среды используется management command `seed_demo_data`. Он создает demo-аккаунты, команды, сервисы и стартовые данные, необходимые для проверки интерфейса. Это упрощает запуск проекта при защите диплома: после применения миграций и seed-команды система сразу готова к демонстрации.

В backend-тестах проверяются основные API-сценарии и доменная логика: регистрация, управление участниками, ограничения размера команды, отправка флагов, checker tick и агрегированные payload-ы. Наличие таких тестов особенно важно для платформы соревнований, потому что ошибки в проверке флага или подсчете баллов напрямую влияют на результат команд.
