# 3.3 Реализация frontend-части на Vue 3

Frontend-часть реализована на Vue 3 с использованием Vite. Приложение является single-page application и взаимодействует с backend через REST API. Такой подход позволяет быстро обновлять состояние соревнования без полной перезагрузки страниц и разделить серверную бизнес-логику от клиентского интерфейса.

Основной shell приложения находится в `frontend/src/App.vue`. Он содержит верхнюю навигацию, переключатель языка, блок текущей сессии, область уведомлений и `router-view`, в котором отображаются route-level страницы. В `App.vue` создается общий реактивный объект `page` на основе composable `useCompetitionPage()`. Затем этот объект передается вниз по дереву компонентов через `provideCompetitionPage`, поэтому отдельные страницы не создают собственные копии состояния, а используют единый контекст соревнования. Это важно для SPA: после входа, выхода, отправки флага или административного действия все route-level страницы работают с согласованным состоянием.

Маршрутизация реализована через `vue-router` в файле `frontend/src/router/index.js`. В текущей версии доступны пять основных маршрутов:

- `/` — dashboard соревнования;
- `/scoreboard` — полная таблица результатов;
- `/services` — матрица состояния сервисов;
- `/team` — командный портал;
- `/admin` — панель организатора.

Каждый маршрут связан с отдельным компонентом из `frontend/src/pages/`. Route-level страницы отвечают за композицию крупных блоков интерфейса, но не содержат сложной бизнес-логики. Например, `DashboardPage.vue` собирает hero-блок, карточки метрик, таблицу результатов, timeline checker-а и историю отправок, а данные для них получает из общего page context. Такое разделение позволяет расширять интерфейс без смешивания сетевых запросов, правил авторизации и верстки в одном компоненте.

Взаимодействие с backend API вынесено в `frontend/src/api.js`. Базовый адрес API задается переменной `VITE_API_BASE_URL`, а по умолчанию используется `http://localhost:8000/api`. Функция `apiRequest` нормализует путь, добавляет заголовки `Accept`, `Content-Type`, `Accept-Language` и при необходимости `Authorization: Token <token>`. Token хранится в `localStorage` под ключом `baltctf_api_token`, что позволяет восстановить сессию после перезагрузки страницы. Если backend возвращает ошибку, `apiRequest` создает исключение с HTTP-статусом и JSON payload-ом, а composable уже преобразует его в пользовательское сообщение.

Центральным элементом frontend-архитектуры является composable `frontend/src/composables/useCompetitionPage.js`. Он выступает orchestration entrypoint и собирает несколько специализированных модулей:

- `competitionPageState.js` — реактивное состояние, формы, loading-флаги и сообщения;
- `competitionPageFactories.js` — фабрики пустых payload-ов и форм;
- `competitionPageMutations.js` — функции синхронизации состояния после ответов backend-а;
- `competitionPageDerived.js` — вычисляемые значения для UI;
- `competitionPageLoaders.js` — загрузчики публичных, сессионных и административных данных;
- `competitionPageAuthTeamActions.js` — сценарии входа, регистрации, резервирования, отправки флага и управления командой;
- `competitionPageAdminActions.js` — сценарии staff-панели.

При монтировании composable параллельно загружает dashboard, service status matrix, настройки регистрации и текущую сессию. После восстановления сессии вызывается загрузка admin state: если пользователь не является staff-пользователем, frontend очищает административное состояние и не выполняет лишний защищенный запрос. Такая последовательность позволяет гостю сразу видеть публичное состояние соревнования, игроку — командный контекст, а администратору — полный срез управления.

В `competitionPageDerived.js` сформированы значения, которые удобны непосредственно для отрисовки: карточки summary, название и метаданные текущего раунда, наличие scoreboard, лидер таблицы, последние раунды, признаки `hasTeam`, `isAdmin`, `isCaptain`, наличие данных service matrix, текущий running round в admin-панели, карточки checker status и ограничения на добавление участников. За счет этого Vue-компоненты остаются декларативными: они получают готовые значения и события, а не повторяют предметные расчеты.

Главная страница dashboard отображает сводку соревнования, компактную таблицу результатов, аналитику по сервисам, timeline checker-проверок и историю отправок. Отдельная страница `/scoreboard` показывает полную таблицу результатов с расширенными метриками. Страница `/services` предназначена для просмотра матрицы статусов сервисов и истории проверок по раундам.

Dashboard построен из нескольких доменных компонентов. `DashboardHero` показывает текущий раунд, лидера и кнопку обновления. `SummaryMetrics` отображает набор агрегированных карточек: количество команд, пользователей, сервисов, раундов, атакующие и защитные баллы, accepted/rejected submissions, количество checker-проверок и процент принятых флагов. `CompetitionBoard` совмещает компактный scoreboard, posture сервисов по командам и recent attack feed. `ServiceTimeline` группирует checker history по раундам и сервисам, показывая распределение статусов `up`, `mumble`, `corrupt`, `down` и `unknown`. `SubmissionHistory` выводит последние попытки отправки флагов с командой, сервисом, статусом, баллами и временем.

Страница `/scoreboard` расширяет dashboard-представление. В верхней части она показывает текущий раунд и refresh action, далее выводит top-3 команды в podium-блоке, сводные показатели соревнования и полную таблицу. Для каждой команды отображаются атакующие баллы, защитные баллы, количество принятых флагов, количество проверок, общий результат и service posture. Ниже находится сервисная аналитика: для каждого сервиса показываются generated flags, accepted submissions, defense points, checker checks, uptime percent и распределение статусов. Поэтому отдельная страница scoreboard подходит не только для зрителей, но и для организатора, которому нужно видеть вклад атак и защиты в итоговый результат.

Страница `/services` использует компонент `ServiceStatusMatrix`. Матрица строится по активным командам и сервисам текущего раунда. В каждой ячейке отображается доменный статус сервиса, защитные баллы и диагностическое сообщение checker-а. В верхней части матрицы выводится summary по статусам, а рядом с ней используется тот же `ServiceTimeline`, что и на dashboard. Такое повторное использование компонента важно для единообразия: краткая и подробная страницы показывают одинаковую интерпретацию checker-результатов.

Командный портал `/team` объединяет сценарии регистрации, входа, просмотра команды, управления участниками и отправки флагов. Если пользователь не авторизован, он видит guest access flow. После входа frontend загружает пользовательский контекст и показывает рабочую область команды. Для капитана доступны дополнительные действия по управлению составом.

Неавторизованный пользователь работает с компонентом `GuestAccessSection`. В нем объединены три сценария: вход по существующему аккаунту, резервирование названия команды и регистрация новой команды. Форма регистрации содержит данные команды, reservation token, данные капитана и список дополнительных участников. Кнопка добавления участника ограничивается значением `MAX_TEAM_MEMBERS`, полученным из frontend factory/composable слоя и согласованным с backend-ограничением. Если регистрационное окно закрыто, интерфейс показывает соответствующий текст и блокирует отправку формы.

После авторизации пользователь видит компонент `UserWorkspace`. Он показывает сведения о сессии, команду, роль пользователя, affiliation, contact email, moderation status, состав команды и историю последних отправок. Для капитана отображаются captain tools: обновление профиля команды, добавление игрока, повышение или понижение роли участника, удаление участника. Эти элементы являются удобством интерфейса, но не заменяют backend-проверки: даже если пользователь вручную вызовет событие или отправит HTTP-запрос, сервер повторно проверит роль капитана. В отдельном блоке находится форма отправки флага `BALTCTF{...}`; после успешной отправки frontend очищает поле, обновляет dashboard и заново загружает сессию, чтобы в командной истории появилась новая запись.

Панель `/admin` предназначена для staff-пользователей. В ней организатор может просматривать состояние соревнования, работать с командами, сервисами, заявками и раундами, а также запускать checker tick. Frontend не является источником прав доступа: все административные операции дополнительно проверяются backend-ом.

`AdminPage.vue` сначала отображает состояние загрузки сессии, затем разделяет три случая: гость, авторизованный пользователь без staff-прав и staff-пользователь. Только в последнем случае монтируется `AdminConsole`. Внутри admin console есть навигация по разделам, сводные карточки, recent submissions, настройки регистрации, заявки на резервирование, управление командами, управление сервисами и lifecycle раундов. В разделе регистрации можно открыть или закрыть окно регистрации, включить обязательные reservation token-ы, настроить auto approve и длительность раундов. В разделе reservation организатор может одобрить или отклонить заявку. В разделе teams поддерживаются создание команды, модерация, включение/отключение и удаление. В разделе services поддерживаются создание, включение/отключение и удаление сервисов. В разделе rounds можно создать одиночный planned round, запланировать batch раундов, запустить или завершить раунд, сгенерировать флаги и вручную выполнить checker tick.

Admin actions реализованы через общий helper `runAdminAction`. Он отправляет `POST`-запрос с token-ом, применяет свежий `admin_state` из ответа, показывает сообщение и обновляет dashboard, registration settings, а при необходимости service matrix. Благодаря этому после административного действия интерфейс сразу отражает новые данные: например, запуск раунда обновляет scoreboard и матрицу сервисов, а checker tick обновляет состояние сервисов и последние checker timestamps.

Интерфейс поддерживает два языка: русский и английский. Словарь находится в `frontend/src/i18n.js`, выбранный язык сохраняется в `localStorage`, а frontend отправляет его в заголовке `Accept-Language`. Это позволяет постепенно развивать серверную локализацию без изменения клиентской архитектуры.

Локализация реализована простым словарным подходом. В `i18n.js` хранится реактивное значение `currentLanguage`, список `languageOptions`, функция `setLanguage`, функция `t` для подстановки строк и форматирование даты с учетом выбранного языка. В словарь вынесены навигация, заголовки страниц, состояния загрузки и ошибок, названия ролей, статусы раундов, статусы сервисов, статусы отправок, тексты team portal и admin console. Такой способ подходит для учебного проекта: он не требует отдельной i18n-библиотеки, но уже дисциплинирует добавление пользовательского текста в одном месте. Кроме того, отправка `Accept-Language` создает основу для будущей локализации backend validation messages.

Стили организованы по SCSS 7-1 architecture. В проекте используются слои `abstracts`, `base`, `components`, `layout`, `pages`, `themes` и `vendors`. Для классов применяется BEM-подобное именование. Такой подход облегчает сопровождение интерфейса, поскольку стили компонентов, страниц и базовых элементов разделены по назначению.

Главный файл `frontend/src/styles/main.scss` подключает слои в порядке `vendors`, `themes`, `base`, `layout`, `components`, `pages`. В `vendors` находится локальная нормализация стилей, в `abstracts` — переменные, карты и mixin-ы, в `base` — reset, типографика и анимации, в `layout` — общий shell, в `components` — стили повторно используемых блоков, а в `pages` — стили route-level страниц. Компонентные SCSS-файлы соответствуют BEM-блокам: `app-shell`, `panel-card`, `guest-access`, `workspace`, `admin-console`, `board`, `service-status`, `round-timeline`, `history`, `state-panel` и другим. Это делает CSS-поверхность предсказуемой и уменьшает риск конфликтов при расширении интерфейса.

Frontend не использует внешнюю UI-библиотеку. Все элементы управления, формы, таблицы, карточки, status tags и state panels реализованы внутри проекта. Для дипломной работы это полезно, потому что позволяет показать собственную структуру интерфейса и связь UI с предметной моделью Attack/Defense CTF, а не только сборку готовых компонентов.

Для frontend-тестирования используется Vitest, `@vue/test-utils` и `jsdom`. В проекте добавлены тесты для app shell, страниц и composable-логики. Это помогает проверять, что route-level страницы корректно монтируются, а общая логика загрузки и изменения состояния не ломается при дальнейших изменениях.

В результате frontend-часть представляет собой не набор независимых страниц, а единое SPA-приложение с общим реактивным контекстом соревнования. Backend остается источником истины для прав доступа, scoring и агрегатов, а frontend отвечает за удобное отображение этих данных для разных ролей: гостя, игрока, капитана и организатора.
