# 2.3 Проектирование модели данных

Модель данных платформы отражает основные сущности Attack/Defense CTF. Центральной сущностью является команда. Команда имеет название, slug, контактный email, принадлежность к организации, статус активности и статус модерации. Статус модерации позволяет организаторам управлять допуском команды к соревнованию.

Участники команды представлены через модель `TeamMember`, связанную со стандартной моделью пользователя Django. Участник имеет роль `captain` или `player`. Капитан получает дополнительные права на управление составом команды. Такой подход позволяет использовать встроенные механизмы Django для пользователей и паролей, но при этом хранить предметную информацию о командной принадлежности отдельно.

Для управления процессом регистрации используется модель `CompetitionSettings`. Она содержит признаки открытия регистрации, временные границы окна регистрации, необходимость предварительного резервирования названия команды, автоматическое одобрение регистраций и параметры длительности раундов. Singleton-подход к настройкам упрощает работу администратора: в системе существует один актуальный набор параметров соревнования.

Модель `TeamReservation` реализует сценарий предварительного резервирования названия команды. Заявка содержит название, slug, контактный email, имя капитана, token, статус и служебные поля. Статусы `pending`, `approved`, `rejected` и `claimed` позволяют организатору контролировать переход от заявки к зарегистрированной команде.

Игровая часть платформы основана на моделях `Service`, `Round`, `Flag`, `Submission` и `ServiceStatus`.

`Service` описывает сервис соревнования: название, slug, описание, порт и признак активности. Благодаря slug backend выбирает соответствующий checker module. Это позволяет добавлять новые сервисы без изменения общей логики раундов.

`Round` хранит номер раунда, состояние и временные отметки. Состояния `planned`, `running` и `finished` отражают жизненный цикл раунда. В каждый момент времени checker ориентируется на текущий running round.

`Flag` связывает команду, сервис и раунд. Для этой тройки действует ограничение уникальности, поэтому в одном раунде для одного сервиса и одной команды существует только один актуальный флаг. Дополнительно у флага может быть срок действия.

`Submission` фиксирует попытку отправки флага. В записи сохраняется атакующая команда, пользователь, отправленное значение, найденный флаг, статус обработки, начисленные баллы, сообщение и временные отметки. Такая модель позволяет вести историю как успешных, так и отклоненных отправок.

`ServiceStatus` хранит результат проверки доступности сервиса команды в конкретном раунде. Статус может принимать значения `up`, `mumble`, `corrupt` и `down`. Для каждой комбинации команда-сервис-раунд действует ограничение уникальности. На основе этих данных рассчитываются защитные баллы и строится матрица состояния сервисов.

Проектируемая модель данных покрывает основной цикл Attack/Defense соревнования и оставляет пространство для расширения. Например, в будущем можно добавить отдельные модели для правил scoring, очередей checker-заданий, per-team инфраструктуры, логов эксплуатации и более подробных событий соревнования.
