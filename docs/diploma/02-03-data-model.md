# 2.3 Проектирование модели данных

Модель данных платформы отражает основные сущности Attack/Defense CTF и разделяет их на три группы: регистрационные данные, игровые данные и результаты проверок. Такая структура позволяет хранить историю действий участников, поддерживать ограничения целостности и строить scoreboard без отдельной таблицы с итоговыми результатами.

Центральной сущностью является `Team`. Команда имеет человекочитаемое название, уникальный `slug`, принадлежность к организации, контактный email, признак активности и статус модерации. Уникальность `name` и `slug` защищает систему от конфликтов при регистрации и при построении адресов demo-сервисов. Поле `moderation_status` принимает значения `approved`, `pending` и `suspended`; через него организатор может допускать команду к игре, оставлять заявку на проверке или временно исключать команду из игрового процесса. Метод `is_approved` используется как доменный признак того, что команда может участвовать в игровых сценариях.

Участники команды представлены моделью `TeamMember`, связанной со стандартной моделью пользователя Django. Связь с пользователем реализована как `OneToOneField`, поэтому один пользователь может состоять только в одной команде. Связь с `Team` является отношением "многие к одному": команда включает нескольких участников, а каждая запись `TeamMember` относится к одной команде. Роль участника хранится в поле `role` и принимает значения `captain` или `player`. Капитан получает дополнительные права на изменение профиля команды и управление составом. На уровне модели также задано ограничение `unique_user_in_team` для пары `team` и `user`; вместе с `OneToOneField` оно фиксирует предметное правило: пользователь не должен дублироваться в составе команды.

Для управления процессом регистрации используется модель `CompetitionSettings`. Она содержит флаг открытия регистрации, временные границы окна регистрации, требование предварительного резервирования названия команды, режим автоматического одобрения регистраций, длительность раунда и длительность перерыва между раундами. В реализации используется singleton-подход: метод `get_solo` возвращает первую запись настроек или создает ее при отсутствии. Это упрощает административный интерфейс, потому что организатор работает с одним актуальным набором параметров соревнования. Метод `is_registration_available` инкапсулирует проверку окна регистрации и используется API регистрации и резервирования.

Модель `TeamReservation` реализует сценарий предварительного резервирования названия команды. Заявка содержит уникальные `name`, `slug` и `token`, контактный email, имя капитана, статус, примечание организатора, срок действия и временные отметки создания и рассмотрения. Статусы `pending`, `approved`, `rejected` и `claimed` описывают полный жизненный цикл заявки: создание, одобрение, отказ и использование при регистрации. При включенном режиме `reservation_required_for_registration` backend проверяет token, соответствие названия и slug, срок действия и статус `approved`; после успешной регистрации заявка переводится в `claimed`.

Игровая часть платформы основана на моделях `Service`, `Round`, `Flag`, `Submission` и `ServiceStatus`. Эти сущности отражают основной цикл Attack/Defense CTF: раунд запускается, для каждой команды и каждого сервиса создается флаг, checker размещает и проверяет флаги, команды атакуют чужие сервисы и отправляют найденные значения.

`Service` описывает vulnerable service соревнования. В модели хранятся название, уникальный `slug`, описание, порт, признак активности и дата создания. `slug` выполняет две функции: используется как стабильный технический идентификатор сервиса и позволяет backend выбрать service-specific checker module. В текущей версии настроены `atlas-board`, `signal-api` и `cold-storage`; при добавлении нового сервиса общая логика раундов может остаться прежней, если для нового `slug` добавлен checker module.

`Round` хранит номер раунда, состояние, время старта, время завершения и дату создания. Номер раунда уникален, что важно для генерации флагов, отображения timeline и сортировки истории. Состояния `planned`, `running` и `finished` отражают жизненный цикл раунда. В каждый момент checker tick работает только с текущим `running` round. При запуске раунда backend обновляет `started_at`, очищает `finished_at` и генерирует недостающие флаги. При завершении раунда backend переводит его в `finished`, записывает `finished_at` и проставляет срок действия флагам без явного `expires_at`.

`Flag` связывает команду-защитника, сервис и раунд. Значение флага уникально во всей системе, а для тройки `team`, `service`, `round` задано ограничение `unique_flag_per_team_service_round`. Это означает, что в одном раунде у одной команды на одном сервисе существует только один актуальный флаг. Такое ограничение критично для корректного scoring: без него checker мог бы создать несколько флагов для одной и той же цели, а отправка флага стала бы неоднозначной. Срок действия хранится в поле `expires_at`; он позволяет отклонять флаги завершенных или устаревших раундов.

`Submission` фиксирует каждую попытку отправки флага. В записи сохраняется атакующая команда (`submitting_team`), пользователь-отправитель (`submitted_by`), найденный `Flag`, исходное отправленное значение, статус обработки, начисленные баллы, сообщение, время отправки и время обработки. Связь с `Flag` и пользователем допускает `NULL`, чтобы история попыток не терялась при удалении связанных объектов или при неизвестном флаге. В отличие от `Flag` и `ServiceStatus`, для `Submission` не задано жесткое ограничение уникальности: системе важно хранить и успешные, и отклоненные попытки. Повторный успешный прием одного и того же флага предотвращается прикладной логикой в транзакции: backend блокирует найденный `Flag`, проверяет наличие accepted submission и только затем создает новую запись.

`ServiceStatus` хранит результат проверки сервиса команды в конкретном раунде. Статус принимает значения `up`, `mumble`, `corrupt` и `down`. Для каждой комбинации `team`, `service`, `round` действует ограничение `unique_service_status_per_round`, поэтому в таблице хранится один актуальный результат проверки этой пары в рамках раунда. При повторном checker tick запись обновляется через `update_or_create`: меняются статус, сообщение, защитные баллы и `reported_at`. Метод `suggested_points` задает текущую шкалу защитных баллов: `up` дает 10 баллов, `mumble` — 5, `corrupt` — 2, `down` — 0.

Основные связи модели данных можно описать следующим образом:

| Сущность | Основные связи | Назначение в Attack/Defense CTF |
| --- | --- | --- |
| `Team` | имеет много `TeamMember`, `Flag`, `Submission`, `ServiceStatus` | Команда одновременно является атакующим субъектом и защитником своих сервисов |
| `TeamMember` | связан с `User` один-к-одному и с `Team` многие-к-одному | Хранит командную роль пользователя и права капитана |
| `Service` | имеет много `Flag` и `ServiceStatus` | Описывает vulnerable service и связывает его с checker module |
| `Round` | имеет много `Flag` и `ServiceStatus` | Делит игру на дискретные временные интервалы |
| `Flag` | принадлежит `Team`, `Service`, `Round`; используется в `Submission` | Является целью атаки и единицей начисления атакующих баллов |
| `Submission` | принадлежит атакующей `Team`, пользователю и найденному `Flag` | Хранит историю отправок и результат проверки флага |
| `ServiceStatus` | связывает `Team`, `Service`, `Round` | Хранит результат checker-а и источник защитных баллов |

На основе этих таблиц backend строит агрегированные payload-ы для dashboard, scoreboard и service matrix. Атакующие баллы суммируются по accepted `Submission`, защитные — по `ServiceStatus`, статистика сервисов строится по `Flag`, `Submission` и статусам проверок, а текущий раунд определяется по состоянию `Round`. Таким образом, модель данных играет роль единого источника истины: итоговые таблицы не сохраняются отдельно, а вычисляются из доменных событий и проверок.

Проектируемая модель покрывает основной цикл Attack/Defense соревнования и оставляет пространство для расширения. В дальнейшем можно добавить отдельные модели для правил scoring, очередей checker-заданий, per-team инфраструктуры, логов эксплуатации, событий аудита, SLA по сервисам и более подробной истории нескольких checker tick внутри одного раунда.
