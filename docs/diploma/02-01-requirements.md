# ГЛАВА 2. ПРОЕКТИРОВАНИЕ ПЛАТФОРМЫ

## 2.1 Требования к платформе

На основе анализа формата Attack/Defense CTF были сформированы требования к платформе. Система должна поддерживать несколько групп пользователей: участников команд, капитанов команд и организаторов соревнования. Для каждой группы требуется отдельный набор сценариев.

К функциональным требованиям относятся:

- регистрация команд и участников;
- аутентификация пользователей по логину и паролю;
- резервирование названия команды и модерация заявок;
- управление составом команды капитаном;
- создание и редактирование сервисов соревнования;
- планирование, запуск и завершение раундов;
- генерация флагов для комбинаций команда-сервис-раунд;
- отправка найденных флагов через веб-интерфейс;
- проверка дубликатов, срока жизни флага и принадлежности флага другой команде;
- автоматическая проверка доступности сервисов через checker;
- отображение таблицы результатов;
- отображение статусов сервисов и истории checker-проверок;
- предоставление панели администратора для управления соревнованием.

Для корректного проведения соревнования важно разграничение прав доступа. Обычный участник должен иметь возможность войти в систему, видеть командный контекст и отправлять флаги. Капитан дополнительно управляет составом своей команды. Администратор управляет настройками регистрации, заявками, командами, сервисами и раундами. Операции администратора должны быть недоступны игрокам.

К нефункциональным требованиям относятся:

- понятный веб-интерфейс для игроков и организаторов;
- адаптивность frontend-приложения;
- возможность локального запуска через Docker Compose;
- использование PostgreSQL в качестве основной базы данных;
- расширяемость модели данных для добавления новых сервисов и правил;
- базовая устойчивость к некорректным пользовательским действиям;
- поддержка двух языков интерфейса: русского и английского;
- наличие тестов для ключевых frontend- и backend-сценариев.

Отдельное требование связано с воспроизводимостью демонстрации. Дипломный проект должен запускаться на локальной машине без ручной настройки множества внешних компонентов. Поэтому в состав репозитория входят `docker-compose.yml`, backend, frontend, checker и демонстрационные vulnbox-сервисы. После применения миграций и загрузки демонстрационных данных пользователь получает готовую среду с тестовыми командами, сервисами и учетными записями.

С учетом учебного характера проекта требования к промышленной эксплуатации ограничены. В текущей версии не требуется высокая отказоустойчивость, горизонтальное масштабирование и полноценная изоляция каждой команды на отдельной виртуальной машине. Эти характеристики рассматриваются как перспективы развития после реализации базового функционального цикла.
