# 2.4 Проектирование API и пользовательских сценариев

API платформы построено вокруг сценариев игроков, капитанов и администраторов. Все endpoint-ы расположены относительно префикса `/api/`. Для аутентификации используется token auth: клиент передает заголовок `Authorization: Token <token>`. Такой способ прост для реализации в учебном проекте и хорошо подходит для SPA-приложения.

Публичные endpoint-ы позволяют получить сводную информацию о соревновании без авторизации. К ним относятся healthcheck, dashboard, scoreboard, service status и настройки регистрации. Эти данные нужны как гостям, так и авторизованным участникам. Например, scoreboard должен быть доступен для просмотра хода соревнования, а настройки регистрации позволяют frontend понять, можно ли сейчас создавать команду.

Сценарий регистрации начинается с получения настроек регистрации. Если требуется резервирование названия команды, пользователь отправляет заявку через endpoint создания reservation. После одобрения заявки команда может пройти регистрацию. При регистрации создается команда, капитан и, при необходимости, дополнительные участники. После входа пользователь получает token и может работать с командным кабинетом.

Сценарий капитана включает управление командой. Капитан может обновить affiliation и contact email команды, добавить нового участника, изменить роль участника или удалить его из команды. Эти операции требуют проверки, что текущий пользователь действительно является капитаном соответствующей команды.

Сценарий отправки флага является одним из ключевых. Пользователь отправляет найденное значение через endpoint `/submit-flag/`. Backend ищет соответствующий флаг, проверяет, что он не принадлежит собственной команде отправителя, не истек по времени и не был ранее успешно сдан этой же командой. При успешной проверке создается accepted submission и начисляются атакующие баллы. При нарушении правил создается rejected submission с поясняющим сообщением.

Административный API объединен под префиксом `/admin/`. Он доступен только staff-пользователям. Администратор может получить общий state, изменить настройки соревнования, одобрять и отклонять заявки на резервирование, создавать и редактировать команды, управлять сервисами, планировать раунды, запускать и завершать раунд, генерировать флаги и запускать checker tick.

Отдельное значение имеют агрегированные payload-ы. Endpoint `/dashboard/` возвращает данные для главной страницы: сводку, scoreboard, аналитику по сервисам, историю checker-проверок и историю отправок. Endpoint `/scoreboard/` возвращает расширенную таблицу результатов. Endpoint `/service-status/` возвращает текущую матрицу сервисов и timeline по раундам. Такой подход уменьшает количество запросов frontend и делает интерфейс проще.

При проектировании API учитывалось, что frontend поддерживает русский и английский языки. Клиент отправляет заголовок `Accept-Language`, чтобы серверная локализация могла развиваться согласованно с интерфейсом. В текущей версии основная локализация выполнена на frontend, а часть backend-сообщений остается англоязычной.
