# 2.4 Проектирование API и пользовательских сценариев

API платформы построено вокруг сценариев игроков, капитанов и администраторов. Все маршруты API расположены относительно префикса `/api/`. Для аутентификации используется token auth: клиент передает заголовок `Authorization: Token <token>`. Такой способ прост для реализации в учебном проекте и хорошо подходит для SPA-приложения.

При проектировании API использован сценарный подход. Вместо набора изолированных CRUD-операций API сгруппировано вокруг реальных действий пользователей: просмотр состояния игры, регистрация, работа с командой, отправка флага, администрирование раундов и запуск checker-а. Это упрощает frontend-часть: страницы получают готовые JSON-ответы, соответствующие экрану и роли пользователя.

Публичные маршруты API позволяют получить сводную информацию о соревновании без авторизации. К ним относятся `GET /health/`, `GET /summary/`, `GET /dashboard/`, `GET /scoreboard/`, `GET /service-status/` и `GET /registration/settings/`. Эти данные нужны гостям, игрокам и администраторам. Например, таблица результатов должна быть доступна для просмотра хода соревнования, а настройки регистрации позволяют frontend понять, можно ли сейчас создавать команду.

Маршрут `/dashboard/` возвращает самый полный публичный срез: текущий раунд, summary, компактную таблицу результатов, аналитику по сервисам, историю раундов, временную шкалу checker-проверок и историю отправок. Маршрут `/scoreboard/` ориентирован на полную таблицу результатов и включает те же расчетные показатели, которые нужны отдельной странице scoreboard. Маршрут `/service-status/` возвращает матрицу состояния сервисов текущего раунда и историю статусов по раундам. Благодаря этим агрегированным ответам frontend не выполняет десятки мелких запросов и не дублирует сложные вычисления начисления баллов.

Сценарий регистрации начинается с получения настроек регистрации. Если требуется резервирование названия команды, пользователь отправляет заявку через маршрут создания reservation. После одобрения заявки команда может пройти регистрацию. При регистрации создается команда, капитан и, при необходимости, дополнительные участники. После входа пользователь получает token и может работать с командным кабинетом.

Более подробно регистрационный поток выглядит так:

1. Frontend запрашивает `GET /registration/settings/` и определяет, открыта ли регистрация.
2. Если включено резервирование, пользователь отправляет `POST /team-reservations/` с названием команды, slug, email и username капитана.
3. Организатор одобряет заявку через административный маршрут API.
4. Пользователь отправляет `POST /auth/register/`, передавая данные команды, капитана, участников и reservation token.
5. Backend создает `Team`, пользователей, `TeamMember`, при необходимости переводит reservation в `claimed` и возвращает token капитана.

Отдельная группа auth-маршрутов обслуживает вход и текущий пользовательский контекст. `POST /auth/login/` выдает token, `POST /auth/logout/` удаляет текущий token, а `GET /auth/me/` возвращает признак аутентификации, пользователя, команду, роль, состав команды и последние отправки. Этот маршрут используется frontend-приложением при открытии командного портала и после командных действий, чтобы синхронизировать интерфейс с серверным состоянием.

Сценарий капитана включает управление командой. Капитан может обновить affiliation и contact email команды, добавить нового участника, изменить роль участника или удалить его из команды. Эти операции требуют проверки, что текущий пользователь действительно является капитаном соответствующей команды.

Капитанский API включает `POST /team/update/`, `POST /team/members/`, `POST /team/members/<user_id>/role/` и `POST /team/members/<user_id>/remove/`. Backend проверяет token, наличие членства в команде и роль `captain`. Для управления составом дополнительно проверяется максимальный размер команды, корректность роли и наличие хотя бы одного капитана после изменения. Пользователь не может удалить собственную учетную запись через маршрут удаления участника, что предотвращает случайную потерю административного доступа к команде.

Обычный игрок работает с более ограниченным набором сценариев. Он может войти в систему, получить контекст команды, просматривать dashboard, scoreboard и service matrix, а также отправлять флаги. Игрок не может менять состав команды или административные настройки. Такое разделение важно для дипломного проекта, потому что оно показывает применение role-based access control не только для staff-панели, но и внутри командного пространства.

Сценарий отправки флага является одним из ключевых. Пользователь отправляет найденное значение через маршрут `/submit-flag/`. Backend ищет соответствующий флаг, проверяет, что он не принадлежит собственной команде отправителя, относится к текущему раунду, не истек по времени и не был ранее успешно принят системой. При успешной проверке создается accepted submission и начисляются атакующие баллы. При нарушении правил создается rejected submission с поясняющим сообщением.

В текущей реализации отправка флага дополнительно ограничена активным running round и лимитом частоты: команда может выполнить не более десяти отправок за минуту. Backend сохраняет rejected submission даже при неизвестном, собственном, просроченном, не относящемся к текущему раунду или уже принятом флаге. Это полезно для интерфейса и анализа действий участников. После accepted submission backend возвращает обновленную таблицу результатов, чтобы frontend мог сразу показать изменение результата.

Административный API объединен под префиксом `/admin/`. Он доступен только staff-пользователям. Администратор может получить общий state, изменить настройки соревнования, одобрять и отклонять заявки на резервирование, создавать и редактировать команды, управлять сервисами, планировать раунды, запускать и завершать раунд, генерировать флаги и запускать checker tick.

Сценарий администратора можно разделить на несколько групп:

| Группа API | Endpoint-ы | Назначение |
| --- | --- | --- |
| Состояние | `GET /admin/state/` | Единый срез для панели администратора |
| Настройки | `POST /admin/settings/update/` | Управление регистрацией и параметрами раундов |
| Заявки | `POST /admin/reservations/<id>/approve/`, `POST /admin/reservations/<id>/reject/` | Модерация резервирования названий команд |
| Команды | `POST /admin/teams/`, `POST /admin/teams/<id>/update/`, `POST /admin/teams/<id>/delete/` | Создание, изменение и удаление команд |
| Сервисы | `POST /admin/services/`, `POST /admin/services/<id>/update/`, `POST /admin/services/<id>/delete/` | Управление vulnerable services |
| Раунды | `POST /admin/rounds/`, `POST /admin/rounds/schedule/`, `POST /admin/rounds/<id>/start/`, `POST /admin/rounds/<id>/finish/` | Планирование и управление жизненным циклом раундов |
| Флаги и checker | `POST /admin/rounds/<id>/generate-flags/`, `POST /admin/checker/tick/` | Подготовка флагов и запуск проверки сервисов |

Маршрут `/admin/state/` играет для административной панели ту же роль, что `/dashboard/` для публичной части. Он возвращает команды, сервисы, раунды, настройки, заявки, последние отправки и агрегированное состояние, чтобы frontend мог отрисовать панель без ручной сборки данных из множества отдельных запросов.

Отдельного внимания требует сценарий управления раундами. Администратор может создать один раунд или запланировать серию раундов с учетом `round_duration_minutes` и `round_break_minutes`. При запуске раунда backend проверяет, что другой раунд уже не находится в состоянии `running`, обновляет временные поля и вызывает генерацию флагов. При завершении раунда флаги без срока действия получают `expires_at`, равный времени окончания. Это связывает API управления раундами с моделью данных и правилами отправки флагов.

Checker-сценарий также реализован через административный API. Внешний checker-контейнер сначала вызывает `POST /auth/login/`, получает token staff-пользователя, а затем периодически отправляет `POST /admin/checker/tick/`. Backend проверяет наличие running round, генерирует недостающие флаги, запускает checker modules и возвращает summary: номер раунда, число созданных флагов, число обработанных статусов и breakdown по `up`, `mumble`, `corrupt`, `down`. Если running round отсутствует, маршрут возвращает конфликтное состояние, а checker остается в режиме ожидания.

Отдельное значение имеют агрегированные JSON-ответы. Маршрут `/dashboard/` возвращает данные для главной страницы: сводку, таблицу результатов, аналитику по сервисам, историю checker-проверок и историю отправок. Маршрут `/scoreboard/` возвращает расширенную таблицу результатов. Маршрут `/service-status/` возвращает текущую матрицу сервисов и timeline по раундам. Такой подход уменьшает количество запросов frontend и делает интерфейс проще.

При проектировании API учитывалось, что frontend поддерживает русский и английский языки. Клиент отправляет заголовок `Accept-Language`, чтобы серверная локализация могла развиваться согласованно с интерфейсом. В текущей версии основная локализация выполнена на frontend, а часть backend-сообщений остается англоязычной.

С точки зрения безопасности все операции, изменяющие состояние, выполняются через `POST` и проходят серверную валидацию. Frontend не является доверенным источником прав: даже если пользователь вручную отправит запрос к административному или капитанскому маршруту, backend повторно проверит token, staff-статус или роль капитана. Для учебной Attack/Defense платформы это особенно важно, потому что участники мотивированы исследовать систему и искать способы обхода ограничений.
