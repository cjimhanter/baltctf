# 2.2 Разработка общей архитектуры системы

Архитектура платформы построена как набор взаимодействующих подсистем: backend, frontend, база данных, checker и демонстрационные vulnbox-сервисы. Такое разделение позволяет независимо развивать пользовательский интерфейс, серверную бизнес-логику и механизм проверки сервисов.

Backend реализован на Django и Django REST Framework. Он отвечает за хранение данных, обработку API-запросов, аутентификацию, проверку флагов, расчет агрегированных показателей и выполнение административных операций. Backend является центральной доверенной частью системы: именно здесь проверяются права пользователей, валидируются отправки флагов и формируются данные для scoreboard.

В качестве базы данных используется PostgreSQL. В ней хранятся команды, участники, настройки соревнования, заявки на резервирование названия команды, сервисы, раунды, флаги, отправки и результаты checker-проверок. Использование реляционной базы данных подходит для данной предметной области, поскольку основные сущности имеют четкие связи и ограничения уникальности.

Frontend реализован на Vue 3 и Vite. Он предоставляет web-интерфейс для игроков и организаторов. Игроки могут зарегистрировать команду, войти в систему, посмотреть scoreboard, проверить состояние сервисов и отправить флаг. Администраторы получают доступ к панели управления, где можно менять настройки соревнования, работать с командами, сервисами и раундами.

Checker является отдельным сервисом, который периодически авторизуется в backend под staff-учетной записью и вызывает endpoint запуска проверки. В текущей реализации backend после такого запроса генерирует недостающие флаги и вызывает service-specific checker modules. Эти модули выполняют реальные HTTP-операции `put` и `get` против demo vulnbox-сервисов. Результаты сохраняются в таблице статусов сервисов.

Vulnbox-сервисы представляют собой намеренно уязвимые демонстрационные приложения. Они нужны для воспроизведения Attack/Defense сценария: checker размещает в них флаги, а участники могут исследовать сервисы и искать способы получения чужих данных. В текущей версии используются три сервиса: `atlas-board`, `signal-api` и `cold-storage`.

Общий поток взаимодействия выглядит следующим образом:

1. Администратор создает или запускает раунд.
2. Backend генерирует флаги для активных команд и сервисов.
3. Checker инициирует проверку текущего раунда.
4. Backend вызывает checker modules для каждого сервиса.
5. Checker modules размещают флаги в demo vulnbox-сервисах и проверяют возможность их чтения.
6. Backend сохраняет результаты в `ServiceStatus`.
7. Команды отправляют найденные флаги через frontend.
8. Backend проверяет отправки и начисляет баллы.
9. Frontend отображает dashboard, scoreboard, матрицу сервисов и историю отправок.

Такой подход упрощает локальный запуск и демонстрацию проекта. При этом архитектура сохраняет возможность дальнейшего развития: checker-логику можно вынести в отдельную очередь задач, demo vulnbox-сервисы заменить на изолированные per-team окружения, а scoring model расширить дополнительными правилами.
