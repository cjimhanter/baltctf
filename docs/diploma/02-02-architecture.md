# 2.2 Разработка общей архитектуры системы

Архитектура платформы построена как набор взаимодействующих подсистем: backend, frontend, база данных, checker и демонстрационные vulnbox-сервисы. Такое разделение позволяет независимо развивать пользовательский интерфейс, серверную бизнес-логику и механизм проверки сервисов.

Общая схема взаимодействия компонентов приведена на рисунке 2.1 в приложении Б. На уровне Docker Compose платформа состоит из сервисов `db`, `backend`, `frontend`, `checker`, `vulnbox-web`, `vulnbox-api` и `vulnbox-storage`. Внешний пользователь работает с frontend-приложением на порту `5173`, frontend обращается к backend API на порту `8000`, backend хранит состояние в PostgreSQL и выполняет HTTP-запросы к demo vulnbox-сервисам внутри docker-сети. Отдельный checker-контейнер не обращается к vulnbox напрямую, а периодически вызывает административный endpoint backend-а.

Backend реализован на Django и Django REST Framework. Он отвечает за хранение данных, обработку API-запросов, аутентификацию, проверку флагов, расчет агрегированных показателей и выполнение административных операций. Backend является центральной доверенной частью системы: именно здесь проверяются права пользователей, валидируются отправки флагов, выбирается текущий running round, создаются флаги и формируются данные для scoreboard. Для аутентификации используется token auth: после входа пользователь получает токен и отправляет его в заголовке `Authorization: Token <token>`.

Backend API условно делится на несколько групп:

- публичные endpoints для проверки состояния, summary, scoreboard, dashboard и service status;
- endpoints регистрации, резервирования названия команды, login/logout и получения текущего пользователя;
- командные endpoints, доступные капитану для изменения профиля команды и состава участников;
- endpoint отправки флага, доступный авторизованным игрокам;
- административные endpoints управления настройками, заявками, командами, сервисами, раундами, генерацией флагов и checker tick.

Такое разделение соответствует ролям предметной области. Игроки и капитаны работают только с командным контекстом, а операции, влияющие на ход соревнования, закрыты проверкой staff-прав. Это важно для Attack/Defense формата: ошибка в разграничении доступа может позволить участникам управлять раундами, создавать флаги или изменять статусы сервисов.

В качестве базы данных используется PostgreSQL. В ней хранятся команды, участники, настройки соревнования, заявки на резервирование названия команды, сервисы, раунды, флаги, отправки и результаты checker-проверок. Использование реляционной базы данных подходит для данной предметной области, поскольку основные сущности имеют четкие связи и ограничения уникальности.

PostgreSQL также используется как единый источник истины для расчетных данных. Например, scoreboard не хранится как отдельная таблица, а строится на основе `Team`, `Submission`, `ServiceStatus`, `Service`, `Round` и связанных агрегатов. Это снижает риск расхождения между историей отправок, checker-статусами и итоговыми баллами. В модели данных предусмотрены ограничения уникальности для команды, сервиса, флага и статуса проверки в рамках раунда, что защищает систему от дублирования критических записей.

Frontend реализован на Vue 3 и Vite. Он предоставляет web-интерфейс для игроков и организаторов. Игроки могут зарегистрировать команду, войти в систему, посмотреть dashboard и полный scoreboard, проверить состояние сервисов и отправить флаг. Администраторы получают доступ к панели управления, где можно менять настройки соревнования, работать с командами, сервисами и раундами.

Frontend-приложение разделено на route-level страницы и переиспользуемые компоненты. Основные маршруты включают dashboard, scoreboard, service matrix, team workspace и admin console. Общий page context формируется через composable `useCompetitionPage.js` и связанные модули состояния, загрузки данных, derived-расчетов и мутаций. Такой подход уменьшает дублирование API-запросов между страницами: dashboard, scoreboard и service matrix получают данные из согласованных backend payload-ов.

Checker является отдельным сервисом, который периодически авторизуется в backend под staff-учетной записью и вызывает endpoint запуска проверки. В текущей реализации backend после такого запроса генерирует недостающие флаги и вызывает service-specific checker modules. Эти модули выполняют реальные HTTP-операции `put` и `get` против demo vulnbox-сервисов. Результаты сохраняются в таблице статусов сервисов.

Checker-контейнер выполняет роль планировщика и изоляционного слоя для периодического запуска проверок. При старте он читает настройки `CHECKER_API_BASE_URL`, `CHECKER_ADMIN_USERNAME`, `CHECKER_ADMIN_PASSWORD`, `CHECKER_INTERVAL_SECONDS` и `CHECKER_REQUEST_TIMEOUT_SECONDS`, выполняет login через backend API и затем с заданным интервалом отправляет `POST /api/admin/checker/tick/`. Если активного running round нет, backend возвращает конфликтное состояние, а checker остается в режиме ожидания. Если токен становится недействительным, checker повторяет аутентификацию.

Внутри backend-а checker tick обрабатывается как единый игровой цикл. Backend проверяет, что существует running round, вызывает генерацию недостающих флагов для всех активных одобренных команд и активных сервисов, затем для каждой пары команда-сервис выбирает checker module по `service.slug`. В текущей версии используются модули для `atlas-board`, `signal-api` и `cold-storage`. Каждый модуль выполняет `put` для размещения флага и `get` для проверки, что флаг можно получить обратно ожидаемым способом. Ошибки преобразуются в доменные статусы `up`, `mumble`, `corrupt` и `down`; каждому статусу соответствует количество защитных баллов.

Vulnbox-сервисы представляют собой намеренно уязвимые демонстрационные приложения. Они нужны для воспроизведения Attack/Defense сценария: checker размещает в них флаги, а участники могут исследовать сервисы и искать способы получения чужих данных. В текущей версии используются три сервиса: `atlas-board`, `signal-api` и `cold-storage`. На уровне Docker Compose они опубликованы на портах `8081`, `8082` и `8083`, а backend обращается к ним по внутренним адресам `vulnbox-web:8080`, `vulnbox-api:8080` и `vulnbox-storage:8080`.

Docker Compose связывает перечисленные компоненты в воспроизводимый локальный стенд. PostgreSQL запускается с healthcheck, backend зависит от готовности базы данных и старта vulnbox-сервисов, frontend зависит от backend-а, а checker зависит от backend-а и базы данных. Переменные окружения позволяют изменить параметры подключения, CORS, адреса vulnbox-сервисов и интервалы checker-а без изменения кода. Такой способ запуска подходит для дипломной демонстрации, потому что все основные части системы поднимаются одной командой и находятся в одном репозитории.

Общий поток взаимодействия выглядит следующим образом:

1. Администратор создает или запускает раунд.
2. Backend генерирует флаги для активных команд и сервисов.
3. Checker инициирует проверку текущего раунда.
4. Backend вызывает checker modules для каждого сервиса.
5. Checker modules размещают флаги в demo vulnbox-сервисах и проверяют возможность их чтения.
6. Backend сохраняет результаты в `ServiceStatus`.
7. Команды отправляют найденные флаги через frontend.
8. Backend проверяет отправки и начисляет баллы.
9. Frontend отображает dashboard, scoreboard, матрицу сервисов и историю отправок.

Для отправки флага игрок использует web-интерфейс frontend-а. Запрос поступает в backend, где проверяется аутентификация пользователя, принадлежность к команде, существование флага, срок его действия, повторная отправка и запрет на отправку собственного флага. Если флаг найден и принадлежит другой команде, создается запись `Submission` со статусом `accepted` и начисляются атакующие баллы. Если проверка не пройдена, попытка также сохраняется как rejected submission, что позволяет отображать историю действий и анализировать ошибки.

Для отображения состояния соревнования frontend использует агрегированные backend payload-ы. Dashboard содержит сводку по командам, сервисам, раундам, атакующим и защитным баллам, компактный scoreboard, timeline checker-проверок и историю отправок. Full scoreboard показывает ранжированный список команд и сервисную аналитику. Service status page отображает матрицу состояния сервисов текущего раунда и историю по раундам. За счет этого пользователю не нужно собирать картину соревнования из множества мелких запросов.

Выбранная архитектура имеет несколько преимуществ для учебного проекта. Во-первых, она явно разделяет пользовательский интерфейс, доменную логику, хранилище данных, планировщик checker-а и vulnerable services. Во-вторых, все компоненты можно запустить локально и проверить без внешней инфраструктуры. В-третьих, backend остается центральной точкой контроля прав и консистентности, что упрощает анализ безопасности и тестирование.

В то же время архитектура содержит осознанные ограничения. Checker tick выполняется синхронно внутри backend request-а, поэтому при большом числе команд и сервисов время ответа будет расти. Demo vulnbox-сервисы в локальной версии общие, а не развернуты отдельно для каждой команды. Отсутствует очередь задач, retry-механика, отдельная сеть команд и полноценное разграничение трафика checker-а и участников. Эти ограничения не противоречат целям дипломной работы, но формируют направления развития: вынесение checker-а в worker-очередь, разворачивание per-team контейнеров, добавление сетевой изоляции, расширение scoring model и создание e2e-тестов для полного цикла раундов.
