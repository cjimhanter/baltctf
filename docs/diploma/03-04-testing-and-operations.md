# 3.4 Тестирование, эксплуатация и ограничения текущей версии

Проверка платформы включает backend-тесты, frontend-тесты и ручные интеграционные сценарии в локальном Docker Compose-контуре. Такой набор соответствует текущему этапу проекта: система уже реализует основной функциональный цикл, но еще не является промышленной платформой для крупных соревнований.

Тестирование в проекте разделено по уровням. Backend-тесты проверяют предметную логику и API-контракты, frontend-тесты проверяют отображение ключевых состояний и работу shared composable-логики, а ручной smoke-сценарий подтверждает, что все контейнеры взаимодействуют между собой в локальной среде. Для дипломного проекта такое разделение важно: оно показывает не только наличие отдельных функций, но и готовность системы к демонстрации полного Attack/Defense цикла.

Backend-тесты запускаются командой:

```bash
docker compose run --rm backend python manage.py test
```

Они проверяют серверную логику приложения Django. Для корректной работы backend также необходимо применять миграции:

```bash
docker compose run --rm backend python manage.py migrate
```

Backend-тесты расположены в `backend/ctf/tests.py` и используют `django.test.TestCase`, тестовый клиент Django и token authentication DRF. Базовый класс `BaseApiTestCase` создает команды, участников, сервис, running round, флаги, отправки, статусы сервисов и staff-пользователя. Это позволяет тестам работать с реалистичным состоянием соревнования, а не с изолированными пустыми таблицами.

Группа `DashboardApiTests` проверяет публичные агрегированные endpoint-ы. Тесты подтверждают, что `/summary/` возвращает количество команд, сервисов, раундов, пользователей, accepted submissions, attack points, defense points, checker statuses и acceptance rate. Для `/scoreboard/` проверяется порядок команд по total points, раздельные attack/defense points, счетчики accepted submissions, defense checks и service breakdown. Для `/dashboard/` проверяются recent activity, submission history, service stats, service status history и round-level метрики. Для `/service-status/` проверяется матрица команда-сервис, summary статусов и история checker-проверок по раундам.

Команда `seed_demo_data` проверяется отдельным тестом. Он запускает команду с `--reset` несколько раз и убеждается, что demo-пользователи, команды, сервисы, раунды и участники создаются идемпотентно. Это важно для защиты диплома: перед демонстрацией можно безопасно перезагрузить demo-данные и получить предсказуемое состояние системы.

Auth-тесты проверяют настройки регистрации, заявки на резервирование имени команды, регистрацию команды с капитаном и участниками, обязательность approved reservation token при включенном режиме резервирования, блокировку регистрации при закрытом окне, вход пользователя и endpoint `/auth/me/`. Отдельно проверяется staff-пользователь без команды: такой пользователь должен получать корректный сессионный payload и иметь возможность работать с admin-панелью.

Admin-тесты покрывают staff-only сценарии: запрет доступа обычному игроку к `/admin/state/`, получение административного snapshot-а, изменение настроек соревнования, пакетное планирование раундов, одобрение и отклонение reservation-заявок, создание и запуск раунда, генерацию флагов, ручной checker tick, обработку отсутствия running round, а также создание, изменение и удаление команд и сервисов. Для checker-а отдельно проверяется dispatcher service-specific модулей и преобразование ошибок checker-а в статусы `up`, `mumble`, `corrupt` и `down`.

Тесты отправки флагов проверяют несколько критичных правил scoring: нельзя отправить флаг своей команды, неизвестный флаг отклоняется, accepted submission начисляет баллы, повторная успешная сдача того же флага не допускается, флаг не из текущего раунда отклоняется, а endpoint требует аутентификации. Такие проверки относятся к наиболее чувствительной части платформы, потому что ошибки здесь напрямую влияют на результат соревнования.

Тесты управления командой проверяют капитанские операции: обновление профиля команды, добавление участника, повышение роли до капитана, удаление участника, запрет этих действий для обычного игрока, запрет удалить собственный аккаунт через team management API, запрет оставить команду без капитана и соблюдение лимита размера команды.

Frontend-тесты запускаются из каталога `frontend`:

```bash
npm run test:run
```

Они используют Vitest и jsdom. Тестами покрыты app shell, route-level страницы и composable-логика, которая управляет состоянием соревнования на клиенте.

Frontend-тесты расположены рядом с исходным кодом: `frontend/src/App.spec.js`, `frontend/src/pages/pages.spec.js`, `frontend/src/composables/useCompetitionPage.spec.js`. Общие тестовые helper-ы находятся в `frontend/src/test/utils.js`, где создаются mock payload-ы dashboard, service status, session, admin state и registration settings. Для сетевых запросов используется управляемый fetch mock, поэтому тесты composable-логики не требуют запущенного backend-а.

`App.spec.js` проверяет shell приложения и переключение языка. Тест монтирует `App.vue` со stub-компонентами `RouterLink`, `RouterView` и `NoticeBanner`, затем переключает язык с English на Русский и проверяет, что навигация обновляется. Этот тест фиксирует работу базового i18n-механизма и реактивность словаря.

`pages.spec.js` проверяет route-level страницы. Для dashboard проверяются loading state, error state, populated scoreboard и refresh action. Для service status page проверяются loading/error состояния, отображение service matrix и checker timeline. Для scoreboard page проверяется полная таблица результатов, service posture, service analytics, latest submissions и refresh action. Для team page проверяются восстановление сессии, гостевой интерфейс, смена языка, authenticated workspace и no-team fallback для operator-аккаунта. Для admin page проверяются блокировка гостя, блокировка обычного пользователя без staff-прав и отображение staff console с переводом текста на русский язык.

`useCompetitionPage.spec.js` проверяет общий orchestration слой. Тесты подтверждают, что при монтировании загружаются public dashboard, service matrix и registration settings; успешный login обновляет сессию и очищает форму; ошибка login отображается без изменения сессии; повторный login-запрос не дублируется, пока первый находится в процессе; отправка флага вызывает `/submit-flag/`, очищает форму и обновляет командную активность; ответ `401` на защищенное действие сбрасывает token и переводит пользователя в гостевое состояние; обновление admin settings синхронизирует admin state и registration settings; ошибка checker tick показывается staff-пользователю как административное сообщение.

Для ручной проверки полного сценария используется локальный запуск через Docker Compose. Типовой порядок следующий: запустить стек, применить миграции, загрузить demo-данные, войти под администратором, создать или запустить раунд, выполнить checker tick, перейти на страницу сервисов и убедиться, что статусы отображаются в матрице. После этого можно войти под demo-игроком и проверить отправку флага.

Локальный контур описан в `docker-compose.yml` и включает сервисы `db`, `backend`, `frontend`, `checker`, `vulnbox-web`, `vulnbox-api` и `vulnbox-storage`. PostgreSQL запускается с healthcheck и volume `db_data`. Backend получает параметры подключения к базе, Django settings, CORS origins и адреса vulnbox-сервисов через переменные окружения. Frontend запускается как Vite-приложение и обращается к backend по `VITE_API_BASE_URL`. Checker-контейнер авторизуется в backend под staff-аккаунтом и периодически вызывает `/api/admin/checker/tick/`. Demo vulnbox-сервисы публикуются на портах `8081`, `8082` и `8083`, а внутри docker-сети доступны backend-у по именам `vulnbox-web`, `vulnbox-api` и `vulnbox-storage`.

Если нужно переопределить значения по умолчанию, сначала создается локальный `.env`:

```bash
cp .env.example .env
```

Основной запуск выполняется командой:

```bash
docker compose up --build
```

Если база данных пустая или нужно подготовить чистую демонстрацию, выполняются миграции и загрузка demo-данных:

```bash
docker compose run --rm backend python manage.py migrate
docker compose run --rm backend python manage.py seed_demo_data --reset
```

Команда `migrate` применяет схему базы данных, а `seed_demo_data --reset` создает демонстрационные команды, участников, сервисы, раунды и учетные записи. После этого frontend сразу получает данные для dashboard, scoreboard, service matrix, team portal и admin console.

Локальные адреса после запуска:

- frontend dashboard: `http://localhost:5173/`;
- scoreboard: `http://localhost:5173/scoreboard`;
- service matrix: `http://localhost:5173/services`;
- team portal: `http://localhost:5173/team`;
- admin tools: `http://localhost:5173/admin`;
- backend API: `http://localhost:8000/api/`;
- Django admin: `http://localhost:8000/admin/`.

К демонстрационным учетным записям относятся staff-пользователь `admin` и несколько demo-игроков, создаваемых командой `seed_demo_data`. Эти данные используются только для локальной демонстрации и не должны применяться в production-среде.

Для защиты ВКР можно использовать следующий ручной smoke-сценарий:

1. Запустить стек через `docker compose up --build`.
2. Применить миграции и загрузить demo-данные.
3. Открыть `http://localhost:5173/` и убедиться, что dashboard показывает команды, сервисы, текущий раунд, таблицу результатов и историю событий.
4. Перейти на `/scoreboard` и показать раздельные attack points, defense points, total points, service posture и service analytics.
5. Перейти на `/team`, войти под demo-игроком, показать состав команды, роль капитана, историю отправок и форму отправки флага.
6. Выйти или открыть новую сессию, войти под `admin` и перейти на `/admin`.
7. В admin console показать настройки регистрации, заявки, команды, сервисы, раунды и recent submissions.
8. Если нет running round, запустить запланированный раунд или создать новый planned round и затем запустить его.
9. Нажать `Run checker tick` или дождаться автоматического цикла checker-контейнера.
10. Перейти на `/services` и проверить, что матрица показывает статусы сервисов по командам, сообщения checker-а и timeline по раундам.
11. Вернуться на `/scoreboard` и показать, что defense points и checker checks учитываются в таблице результатов.
12. Для демонстрации Attack-сценария открыть один из уязвимых demo endpoint-ов vulnbox-сервисов, например leak/debug/raw endpoint, получить флаг другой команды и отправить его через `/team`. После accepted submission scoreboard должен обновить атакующие баллы.

Этот сценарий не заменяет автоматическое e2e-тестирование, но он удобен для защиты диплома: за короткое время демонстрируются роли игрока, капитана и администратора, работа backend API, checker tick, vulnbox-сервисы, scoring и frontend-представления.

Текущая версия имеет несколько ограничений. Во-первых, checker выполняется синхронно внутри backend request-а. При увеличении числа команд и сервисов такой подход может привести к долгим HTTP-запросам и блокировке staff-действия до завершения всех проверок. Более масштабная версия должна использовать очередь задач, отдельные worker-процессы, retry-механику, deadline раунда и хранение подробной истории нескольких checker-попыток.

Во-вторых, demo vulnbox-сервисы являются общими для локального контура, а не изолированными per-team окружениями. Внутри demo-сервисов данные разделяются по `team_slug`, но это не заменяет сетевую и файловую изоляцию, принятую в полноценных Attack/Defense соревнованиях. Следующим шагом может стать генерация отдельных контейнеров или виртуальных машин для каждой команды, ограничение сетевого доступа и отдельная маршрутизация checker-а.

В-третьих, demo vulnbox-сервисы хранят данные в памяти процесса. Это упрощает демонстрацию и повторный запуск, но не подходит для длительных соревнований, где требуется устойчивость к перезапуску, аудит действий и восстановление состояния. Для production-подобной версии потребуется постоянное хранилище или контролируемая модель ephemeral-сервисов.

В-четвертых, часть backend validation messages пока остается англоязычной, хотя frontend-интерфейс уже поддерживает русский и английский языки. Frontend отправляет `Accept-Language`, но полноценная серверная локализация еще не реализована. Для финального двуязычного продукта нужно перевести backend-сообщения и унифицировать формат ошибок API.

В-пятых, пока отсутствует полноценный e2e-набор тестов, который поднимает весь Docker-стек и автоматически проверяет checker/vulnbox flow против реальных HTTP-сервисов. Наличие unit/API-тестов и Vitest-тестов снижает риск регрессий, но не покрывает все сетевые и контейнерные взаимодействия. Для дальнейшего развития целесообразно добавить отдельный e2e-пайплайн, который запускает Docker Compose, применяет миграции, seed-ит данные, выполняет checker tick, извлекает флаг через demo-уязвимость и отправляет его в backend.

Также текущая scoring model фиксирована в коде: accepted flag дает заданное количество атакующих баллов, а защитные баллы зависят от статуса сервиса. Для учебного стенда этого достаточно, но для реальных мероприятий может потребоваться конфигурируемая модель начисления, штрафы за недоступность, decay-механика, SLA-агрегаты и экспорт результатов.

Несмотря на эти ограничения, текущая реализация позволяет продемонстрировать основной цикл Attack/Defense CTF: регистрацию команды, запуск раунда, генерацию и проверку флагов, отправку найденных флагов, расчет результатов и отображение состояния соревнования в web-интерфейсе.
