# 3.4 Тестирование, эксплуатация и ограничения текущей версии

Проверка платформы включает backend-тесты, frontend-тесты и ручные интеграционные сценарии в локальном Docker Compose-контуре. Такой набор соответствует текущему этапу проекта: система уже реализует основной функциональный цикл, но еще не является промышленной платформой для крупных соревнований.

Backend-тесты запускаются командой:

```bash
docker compose run --rm backend python manage.py test
```

Они проверяют серверную логику приложения Django. Для корректной работы backend также необходимо применять миграции:

```bash
docker compose run --rm backend python manage.py migrate
```

Frontend-тесты запускаются из каталога `frontend`:

```bash
npm run test:run
```

Они используют Vitest и jsdom. Тестами покрыты app shell, route-level страницы и composable-логика, которая управляет состоянием соревнования на клиенте.

Для ручной проверки полного сценария используется локальный запуск через Docker Compose. Типовой порядок следующий: запустить стек, применить миграции, загрузить demo-данные, войти под администратором, создать или запустить раунд, выполнить checker tick, перейти на страницу сервисов и убедиться, что статусы отображаются в матрице. После этого можно войти под demo-игроком и проверить отправку флага.

Локальные адреса после запуска:

- frontend dashboard: `http://localhost:5173/`;
- scoreboard: `http://localhost:5173/scoreboard`;
- service matrix: `http://localhost:5173/services`;
- team portal: `http://localhost:5173/team`;
- admin tools: `http://localhost:5173/admin`;
- backend API: `http://localhost:8000/api/`;
- Django admin: `http://localhost:8000/admin/`.

К демонстрационным учетным записям относятся staff-пользователь `admin` и несколько demo-игроков, создаваемых командой `seed_demo_data`. Эти данные используются только для локальной демонстрации и не должны применяться в production-среде.

Текущая версия имеет несколько ограничений. Во-первых, checker выполняется синхронно внутри backend request-а. При увеличении числа команд и сервисов такой подход может привести к долгим HTTP-запросам. Во-вторых, demo vulnbox-сервисы являются общими для локального контура, а не изолированными per-team окружениями. В-третьих, часть backend validation messages пока остается англоязычной, хотя frontend-интерфейс уже поддерживает русский и английский языки. В-четвертых, пока отсутствует полноценный e2e-набор тестов, который поднимает весь Docker-стек и проверяет checker/vulnbox flow автоматически.

Несмотря на эти ограничения, текущая реализация позволяет продемонстрировать основной цикл Attack/Defense CTF: регистрацию команды, запуск раунда, генерацию и проверку флагов, отправку найденных флагов, расчет результатов и отображение состояния соревнования в web-интерфейсе.
