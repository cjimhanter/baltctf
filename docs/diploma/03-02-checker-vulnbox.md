# 3.2 Реализация checker system и демонстрационных vulnbox-сервисов

Checker system отвечает за автоматическую проверку сервисов команд. В Attack/Defense CTF checker выполняет две важные функции: размещает флаги в сервисах и проверяет, что сервисы продолжают корректно работать. Если сервис недоступен или не сохраняет ожидаемые данные, команда не должна получать защитные баллы за этот раунд.

В проекте `baltctf` checker представлен отдельным контейнером `checker`. Он запускается вместе с остальными сервисами через Docker Compose, авторизуется в backend под staff-учетной записью и периодически вызывает endpoint `POST /api/admin/checker/tick/`. Если в системе нет активного running round, checker остается в режиме ожидания и логирует причину.

Внешний checker-контейнер реализован в `checker/main.py`. При старте он считывает `CHECKER_API_BASE_URL`, `CHECKER_ADMIN_USERNAME`, `CHECKER_ADMIN_PASSWORD`, `CHECKER_INTERVAL_SECONDS` и `CHECKER_REQUEST_TIMEOUT_SECONDS`. Затем создается `requests.Session`, выполняется login через `/api/auth/login/`, token сохраняется в памяти процесса и используется в заголовке `Authorization: Token <token>`. Если backend возвращает `401` или `403`, checker сбрасывает token и повторяет аутентификацию на следующем цикле. Если backend возвращает `409` из-за отсутствия running round, checker не считает это ошибкой и продолжает ожидание.

В текущей реализации основная checker-логика выполняется backend-ом после получения tick-запроса. Backend определяет текущий раунд, генерирует недостающие флаги для активных команд и сервисов, затем вызывает отдельный checker module в зависимости от `service.slug`. Такой подход упрощает реализацию учебного прототипа: все данные доступны в одной транзакционной системе, а внешний checker выполняет роль планировщика.

Backend-часть checker-а находится в `backend/ctf/competition.py` и `backend/ctf/checkers/`. Функция `run_checker_tick` проходит по всем активным одобренным командам и активным сервисам текущего раунда. Для каждой пары команда-сервис выбирается флаг из таблицы `Flag`; если флаг отсутствует, результат считается `corrupt`, потому что нарушен ожидаемый инвариант контроллера. Если флаг найден, backend вызывает `run_service_checker(service.slug, ...)`, а dispatcher в `checkers/services.py` выбирает конкретный модуль.

Для демонстрации используются три intentionally vulnerable сервиса:

- `atlas-board` — web board service, где checker размещает флаг в публикации;
- `signal-api` — JSON API service, где флаг хранится в record;
- `cold-storage` — текстовое файловое хранилище, где флаг сохраняется в файле.

Каждый checker module выполняет пару операций `put` и `get`. Операция `put` размещает флаг в соответствующем сервисе. Операция `get` проверяет, что ранее размещенный флаг можно получить обратно ожидаемым способом. Если обе операции успешны, сервис считается доступным и получает статус `up`. При ошибках возможны статусы `mumble`, `corrupt` или `down`, отражающие разные виды некорректного поведения.

Для `atlas-board` checker отправляет `POST /teams/<team_slug>/posts` с полями `slot`, `headline` и `body`, где `body` содержит флаг. Затем он запрашивает `GET /teams/<team_slug>/posts/<slot>` и проверяет, что HTML-страница содержит исходное значение флага. Для `signal-api` checker использует заголовок `X-Team-Token`, создает record через `POST /api/teams/<team_slug>/records`, затем читает его через `GET /api/teams/<team_slug>/records/<slot>` и сравнивает поле `secret`. Для `cold-storage` checker отправляет файл через `POST /files/<team_slug>/upload`, затем скачивает его через `GET /files/<team_slug>/download/<filename>` и сравнивает текст ответа с флагом.

Ошибки checker modules приводятся к единой доменной модели. Некорректный HTTP-статус при ожидаемом ответе преобразуется в `mumble`. Если сервис вернул данные, но флаг искажен или отсутствует, выбрасывается `CheckerCorruptError`, и backend сохраняет `corrupt`. Сетевые ошибки `requests` считаются `down`. Если для `service.slug` нет зарегистрированного checker module, сервис также получает `down` с диагностическим сообщением. Благодаря этому frontend и scoring работают с небольшим набором понятных статусов, не завися от деталей конкретного сервиса.

Результаты checker-проверок сохраняются в модели `ServiceStatus`. Для каждой комбинации команда-сервис-раунд хранится один актуальный результат. На основе этих данных backend начисляет защитные баллы и строит матрицу состояния сервисов. Frontend показывает эти данные на странице service status и в dashboard timeline.

Демонстрационные vulnbox-сервисы реализованы на Flask и запускаются отдельными контейнерами `vulnbox-web`, `vulnbox-api` и `vulnbox-storage`. В локальном Docker Compose они опубликованы на портах `8081`, `8082` и `8083`, а backend обращается к ним по внутренним адресам `http://vulnbox-web:8080`, `http://vulnbox-api:8080` и `http://vulnbox-storage:8080`. Каждый сервис хранит данные в памяти процесса, что достаточно для демонстрационного стенда и упрощает повторный запуск.

Сервисы специально содержат уязвимые endpoint-ы. `atlas-board` имеет публичный `/leak-board?team=<slug>`, который возвращает публикации выбранной команды. `signal-api` требует простой заголовок `X-Team-Token` для штатного доступа, но имеет `/api/debug/export?team=<slug>`, раскрывающий записи команды. `cold-storage` позволяет читать данные через `/files/raw/<team>/<path>` без полноценной авторизации. Эти уязвимости нужны для воспроизведения Attack/Defense формата: checker кладет флаги штатным способом, а участники могут исследовать сервисы и искать обходные пути получения чужих флагов.

Важная особенность реализации состоит в том, что checker работает не с mock-данными, а с реальными HTTP-запросами к demo vulnbox-сервисам. Поэтому в дипломной демонстрации можно показать полный цикл: запуск раунда, генерация флагов, размещение флага checker-ом, получение флага через уязвимый endpoint, отправка флага в backend и изменение scoreboard.

Текущая реализация имеет ограничение: checker tick выполняется синхронно внутри backend request-а. Для небольшого учебного стенда это допустимо, но для соревнований с большим количеством команд и сервисов лучше использовать очередь задач, отдельные worker-процессы, retry-механику и deadline для раундов. Это направление указано как дальнейшее развитие платформы.

Еще одно ограничение связано с инфраструктурой demo-сервисов. В текущем локальном стенде они являются общими и разделяют данные по `team_slug` внутри одного процесса, тогда как в промышленном Attack/Defense CTF обычно используется отдельное окружение или отдельный набор контейнеров для каждой команды. Следующим шагом развития может стать генерация per-team vulnbox-контейнеров, изоляция сетей команд, отдельная очередь checker-заданий и хранение подробной истории нескольких проверок внутри одного раунда.
