# План работ

Актуально на **17 апреля 2026**.
Дата сдачи: **15 июня 2026**.

Текущий статус: в репозитории уже есть рабочий вертикальный срез Attack/Defense CTF-платформы. Дальнейший план больше не про сборку MVP с нуля, а про стабилизацию, проверку полного цикла, расширение тестирования, демонстрационный сценарий и финализацию материалов ВКР.

## Что уже выполнено

### 1. Аналитика, требования и архитектура

- [x] Изучена предметная область Attack/Defense CTF.
- [x] Зафиксированы функциональные требования к платформе.
- [x] Спроектирована архитектура: backend, frontend, checker, demo vulnbox-сервисы и PostgreSQL.
- [x] Описана текущая модель данных: команды, участники, настройки соревнования, заявки на резервирование, сервисы, раунды, флаги, сабмиты и статусы сервисов.
- [x] Подготовлены основные технические документы в `docs/`: текущее состояние, архитектура, API, frontend и runbook.

### 2. Backend

- [x] Реализованы Django-модели доменной области соревнования.
- [x] Реализованы регистрация команды, резервирование имени команды, токен-аутентификация, logout и endpoint текущего пользователя.
- [x] Реализовано captain-only управление профилем команды и составом участников.
- [x] Реализована отправка флагов с проверкой дубликатов, срока жизни, текущего раунда и запретом на сдачу собственного флага.
- [x] Реализованы staff-only API для настроек соревнования, модерации команд и резервирований, управления сервисами, планирования/запуска/завершения раундов, генерации флагов и запуска checker tick.
- [x] Расширены payload'ы dashboard, scoreboard и service-status: attack/defense stats, service analytics, recent rounds, checker history и submission history.
- [x] Добавлена диагностика checker-а в `admin/state`: expected/checked/unknown counts, issue count и последние проблемные сообщения.
- [x] Добавлена локализация частых backend validation/error messages при `Accept-Language: ru`.
- [x] Есть базовые backend-тесты для dashboard/scoreboard/service-status, регистрации, auth, admin API, flag submission, team management и checker tick.

### 3. Frontend

- [x] Реализовано Vue 3 + Vite приложение с маршрутизацией.
- [x] Реализованы маршруты `/`, `/scoreboard`, `/services`, `/team` и `/admin`.
- [x] Dashboard показывает summary metrics, scoreboard, timeline раундов, service timeline и submission history.
- [x] Scoreboard вынесен в отдельную страницу с expanded team/service stats.
- [x] Service status page показывает матрицу сервисов и checker timeline.
- [x] Team portal поддерживает регистрацию, вход, профиль команды, управление участниками и отправку флагов.
- [x] Admin console поддерживает операторские действия и показывает checker diagnostics.
- [x] Добавлено переключение языка English / Русский; frontend отправляет `Accept-Language`.
- [x] Стили разложены по SCSS 7-1 architecture.
- [x] Есть базовые Vitest-тесты для app shell, страниц и composables.

### 4. Checker и demo vulnbox

- [x] Реализован отдельный контейнер `checker`, который логинится в backend под staff-учёткой и периодически вызывает `POST /api/admin/checker/tick/`.
- [x] Backend checker tick генерирует недостающие флаги и запускает service-specific checker modules.
- [x] Checker больше не использует deterministic mock-состояние: для demo-сервисов выполняются реальные HTTP `put/get` операции.
- [x] Подготовлены три demo vulnbox-сервиса:
  - [x] `atlas-board`
  - [x] `signal-api`
  - [x] `cold-storage`
- [x] Результаты проверок сохраняются в `ServiceStatus` и отображаются во frontend.

### 5. Локальный стенд и документация

- [x] Подготовлен Docker Compose стенд для backend, frontend, PostgreSQL, checker и demo vulnbox-сервисов.
- [x] Подготовлена команда `seed_demo_data --reset` для воспроизводимых demo-данных.
- [x] В README и runbook описаны адреса UI, API, Django admin и demo-аккаунты.
- [x] Зафиксированы текущие ограничения: синхронный checker внутри backend request-а, общие demo vulnbox-сервисы без изоляции по командам, неполная локализация редких Django-сообщений и отсутствие полноценного Docker e2e набора.
